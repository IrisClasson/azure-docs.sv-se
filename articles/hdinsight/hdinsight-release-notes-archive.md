### <a name="new-features"></a>Nya funktioner

#### <a name="service-tags"></a>Tjänsttaggar
Tjänsttaggar förenklar säkerheten för virtuella Azure-datorer och virtuella Azure-nätverk genom att du enkelt kan begränsa nätverksåtkomsten till Azure-tjänsterna. Du kan använda tjänsttaggar i NSG-reglerna (Network Security Group) för att tillåta eller neka trafik till en viss Azure-tjänst globalt eller per Azure-region. Azure tillhandahåller underhåll av IP-adresser som ligger till grund för varje tagg. HDInsight-tjänsttaggar för nätverkssäkerhetsgrupper (NSG) är grupper av IP-adresser för hälso- och hanteringstjänster. Dessa grupper hjälper till att minimera komplexiteten för att skapa säkerhetsregeln. HDInsight-kunder kan aktivera servicetag via Azure Portal, PowerShell och REST API. Mer information finns i [NSG-tjänsttaggar (Network Security Group) för Azure HDInsight](https://docs.microsoft.com/azure/hdinsight/hdinsight-service-tags).

#### <a name="custom-ambari-db"></a>Anpassad Ambari-databas
HDInsight kan du nu använda din egen SQL DB för Apache Ambari. Du kan konfigurera den här anpassade Ambari DB från Azure-portalen eller via resurshanterarens mall.  Med den här funktionen kan du välja rätt SQL DB för dina bearbetnings- och kapacitetsbehov. Du kan också enkelt uppgradera för att matcha tillväxtkraven för företag. Mer information finns i [Konfigurera HDInsight-kluster med en anpassad Ambari DB](hdinsight-custom-ambari-db.md).

![Anpassad Ambari-databas](./media/hdinsight-release-notes/custom-ambari-db.png)

#### <a name="transport-layer-security-tls-12-enforcement"></a>TLS (Transport Layer Security) 1.2
TLS (Transport Layer Security) och SSL (Secure Sockets Layer) är kryptografiska protokoll som ger kommunikationssäkerhet över ett datornätverk. Mer information finns i [Transport layer security](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0). Azure HDInsight-kluster accepterar TLS 1.2-anslutningar på offentliga HTTPS-slutpunkter, men TLS 1.1 stöds fortfarande för bakåtkompatibilitet med äldre klienter.

Från och med nästa utgåva kan du anmäla dig och konfigurera dina nya HDInsight-kluster så att de bara accepterar TLS 1.2-anslutningar. 

Senare under året, från och med 2020-06-30, kommer Azure HDInsight att framtvinga TLS 1.2 eller senare versioner för alla HTTPS-anslutningar. Vi rekommenderar att du ser till att alla dina klienter är redo att hantera TLS 1.2 eller senare versioner.

### <a name="new-features"></a>Nya funktioner

#### <a name="hdinsight-identity-broker-hib-preview"></a>HDInsight Identity Broker (HIB) (Preview)

HDInsight Identity Broker (HIB) gör det möjligt för användare att logga in på Apache Ambari med multifaktorautentisering (MFA) och få de kerberos-biljetter som krävs utan att behöva lösenordshökar i Azure Active Directory Domain Services (AAD-DS). För närvarande är HIB endast tillgängligt för kluster som distribueras via ARM-mall (Azure Resource Management).

#### <a name="kafka-rest-api-proxy-preview"></a>Kafka Rest API Proxy (Preview)

Kafka Rest API Proxy ger ett klick distribution av högtillgänge REST proxy med Kafka kluster via säkrad AAD auktorisering och OAuth protokoll. 

#### <a name="auto-scale"></a>Autoskala

Automatisk skalning för Azure HDInsight är nu allmänt tillgängligt i alla regioner för Apache Spark- och Hadoop-klustertyper. Den här funktionen gör det möjligt att hantera arbetsbelastningar för stordataanalys på ett mer kostnadseffektivt och produktivt sätt. Nu kan du optimera användningen av dina HDInsight-kluster och bara betala för det du behöver.

Beroende på dina behov kan du välja mellan belastningsbaserad och schemabaserad automatisk skalning. Inläsningsbaserad automatisk skalning kan skala klusterstorleken uppåt och nedåt baserat på de aktuella resursbehoven, medan schemabaserad automatisk skalning kan ändra klusterstorleken baserat på ett fördefinierat schema. 

Stöd för automatisk skalning för HBase- och LLAP-arbetsbelastning är också offentlig förhandsversion. Mer information finns i [Skala Azure HDInsight-kluster automatiskt](https://docs.microsoft.com/azure/hdinsight/hdinsight-autoscale-clusters).

#### <a name="hdinsight-accelerated-writes-for-apache-hbase"></a>HDInsight Accelererad skriver för Apache HBase 

Accelererade skrivningar använder Azure Premium SSD-hanterade diskar för att förbättra prestanda hos Apache HBase Write Ahead Log (WAL). Mer information finns i [Azure HDInsight – Accelererade skrivningar för Apache HBase](https://docs.microsoft.com/azure/hdinsight/hbase/apache-hbase-accelerated-writes).

#### <a name="custom-ambari-db"></a>Anpassad Ambari-databas

HDInsight erbjuder nu en ny kapacitet för att kunderna ska kunna använda sin egen SQL DB för Ambari. Nu kunder kan välja rätt SQL DB för Ambari och enkelt uppgradera den baserat på sin egen verksamhet tillväxtkrav. Distributionen görs med en Azure Resource Manager-mall. Mer information finns i [Konfigurera HDInsight-kluster med en anpassad Ambari DB](https://docs.microsoft.com/azure/hdinsight/hdinsight-custom-ambari-db).

#### <a name="f-series-virtual-machines-are-now-available-with-hdinsight"></a>Virtuella datorer i F-serien finns nu tillgängliga med HDInsight

Virtuella datorer i F-serien är bra val för att komma igång med HDInsight med lätta bearbetningskrav. F-serien har ett lägre listpris per timme och har bästa prisprestanda i Azure-portföljen baserat på Azure-beräkningsenhet (ACU, Azure Compute Unit) per virtuell processor. Mer information finns i [Välja rätt VM-storlek för ditt Azure HDInsight-kluster](https://docs.microsoft.com/azure/hdinsight/hdinsight-selecting-vm-size).

### <a name="known-issues"></a>Kända problem

#### <a name="replication-is-broken-for-secure-hbase-with-default-installation"></a>Replikeringen är bruten för Secure HBase med standardinstallation

För HDInsight 4.0 gör du följande steg:

1. Aktivera kommunikation mellan kluster.
1. Logga in på den aktiva huvudnoden.
1. Hämta ett skript för att aktivera replikering med följande kommando:

    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Skriv kommandot `sudo kinit <domainuser>`.
1. Skriv följande kommando för att köra skriptet:

    ```
    sudo bash hdi_enable_replication.sh -m <hn0> -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```
För HDInsight 3.6 gör du följande:

1. Logga in på active HMaster ZK.
1. Hämta ett skript för att aktivera replikering med följande kommando:
    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Skriv kommandot `sudo kinit -k -t /etc/security/keytabs/hbase.service.keytab hbase/<FQDN>@<DOMAIN>`.
1. Ange följande kommando:

    ```bash
    sudo bash hdi_enable_replication.sh -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```

1. Återskapa Phoenix-tabeller och -vyer för dina HBase-data.

#### <a name="mahout"></a>Mahout

I HDP-2.3.x och 2.4.x, istället för att skicka en specifik Apache-version av Mahout, synkroniserade vi till en viss revisionspunkt på Apache Mahout-bagageutrymmet. Denna revidering punkt är efter 0.9.0 release, men innan 0.10.0 release. Detta ger ett stort antal buggfixar och funktionella förbättringar under 0.9.0-versionen, men ger en stabil version av Mahout-funktionen innan den fullständiga konverteringen till nya Spark-baserade Mahout i 0.10.0.

Den revisionspunkt som valts för Mahout i HDP 2.3.x och 2.4.x är från "mahout-0.10.x" av Apache Mahout, från och med den 19 december 2014, revision 0f037cb03e77c096 i GitHub.

I HDP-2.5.x och 2.6.x tog vi bort biblioteket "commons-httpclient" från Mahout eftersom vi ser det som ett föråldrat bibliotek med möjliga säkerhetsproblem och uppgraderade Hadoop-Client i Mahout till version 2.7.3, samma version som används i HDP-2.5. Som ett resultat:

-   Tidigare kompilerade Mahout jobb kommer att behöva kompileras i HDP-2.5 eller 2.6 miljö.

-   Det finns en liten möjlighet att vissa Mahout-jobb kan stöta på fel i "ClassNotFoundException" eller "could not load class" relaterade till "org.apache.commons.httpclient", "net.java.dev.jets3t" eller relaterade klassnamnprefix. Om dessa fel inträffar kan du överväga om du manuellt ska installera de nödvändiga burkarna i din klasssökväg för jobbet, om risken för säkerhetsproblem i det föråldrade biblioteket är acceptabel i din miljö.

-   Det finns en ännu mindre möjlighet att vissa Mahout jobb kan stöta på kraschar i Mahout's hbase-klient kod samtal till hadoop-gemensamma bibliotek, på grund av binära kompatibilitetsproblem. Tyvärr finns det inget sätt att lösa problemet förutom återgå till HDP-2.4.2 version av Mahout, som kan ha säkerhetsproblem. Återigen bör detta vara mycket ovanligt, och är osannolikt att inträffa i någon viss Mahout jobbsvit.

  -   [RANGER-1607](https://issues.apache.org/jira/browse/RANGER-1607): Lösning för Ranger-tjänstkontrollfel vid uppgradering till HDP 2.6.2 från tidigare HDP-versioner.

      > [!NOTE]  
      > Endast när Ranger är SSL-aktiverad. 
     
  Det här problemet uppstår när du försöker uppgradera till HDP-2.6.1 från tidigare HDP-versioner via Ambari. Ambari använder ett curl-samtal för att göra en servicekontroll till Ranger-tjänsten i Ambari. Om JDK-versionen som används av Ambari är JDK-1.7, kommer curl-samtalet att misslyckas med felet nedan:
    
  `curl: (35) error:14077410:SSL routines:SSL23_GET_SERVER_HELLO:sslv3 alert handshake failure`
    
  Anledningen till detta fel är tomcat version som används i Ranger är\*Tomcat-7.0.7 . Använda JDK-1.7 konflikter med standard chiffer som tillhandahålls i Tomcat-7.0.7\*.
    
  Du kan lösa problemet på två sätt:
    
  -   Uppdatera JDK som används i Ambari från JDK-1.7 till JDK-1.8 (se avsnittet [Ändra JDK-versionen](https://docs.hortonworks.com/HDPDocuments/Ambari-2.2.0.0/bk_ambari_reference_guide/content/ch_changing_the_jdk_version_on_an_existing_cluster.html) i Ambari-referensguiden).
    
  -   Om du vill fortsätta stödja en JDK-1.7-miljö:
    
      1.  Lägg till egenskapsintervaller.tomcat.chiffer i avsnittet ranger-admin-site i konfigurationen av Ambari Ranger med nedanvärdet:
            
          SSL\_\_RSA\_MED\_RC4\_128\_MD5,\_SSL\_RSA\_\_MED RC4 128\_SHA,\_TLS\_\_RSA\_MED\_\_AES\_128 CBC SHA, SSL\_RSA\_MED 3DES EDE\_CBC\_SHA
        
      2.  Om din miljö är konfigurerad för Ranger-KMS lägger du till egenskapsintervaller.tomcat.chiffer i avsnittetranger-kms-site i konfigurationen av Ambari Ranger med nedanvärdet:
            
          SSL\_\_RSA\_MED\_RC4\_128\_MD5,\_SSL\_RSA\_\_MED RC4 128\_SHA,\_TLS\_\_RSA\_MED\_\_AES\_128 CBC SHA, SSL\_RSA\_MED 3DES EDE\_CBC\_SHA
    
  >[!NOTE]  
  >De noterade värdena är arbetsexempel och kanske inte är ett tecken på din miljö. Kontrollera att det sätt du anger dessa egenskaper matchar hur din miljö är konfigurerad.

### <a name="deprecation"></a>Utfasning

-   **OMS-portal:** Vi har tagit bort länken från HDInsight resurs sida som pekade på OMS portal. Azure Monitor loggar använde ursprungligen sin egen portal som kallas OMS-portalen för att hantera sin konfiguration och analysera insamlade data. Alla funktioner från den här portalen har flyttats till Azure-portalen där den kommer att fortsätta att utvecklas. HDInsight har inaktuellt stöd för OMS-portalen. Kunder använder HDInsight Azure Monitor-loggintegrering i Azure-portalen.

    -   <https://spark.apache.org/releases/spark-release-2-3-0.html#deprecations>

Om du vill hämta den senaste versionen av Spark, Kafka och R Server (Machine Learning Services) väljer du versionen Spark, Kafka, ML Services när du [skapar ett HDInsight 3.6-kluster](https://docs.microsoft.com/azure/hdinsight/hdinsight-hadoop-provision-linux-clusters). Om du vill få stöd för ADLS kan du välja ADLS-lagringstypen som ett alternativ. Befintliga kluster uppgraderas inte automatiskt till dessa versioner.

Alla nya kluster som skapas efter juni 2018 får automatiskt 1000+ buggfixar i alla projekt med öppen källkod. Följ [den här](https://docs.microsoft.com/azure/hdinsight/hdinsight-upgrade-cluster) guiden för bästa praxis kring uppgradering till en nyare HDInsight-version.
